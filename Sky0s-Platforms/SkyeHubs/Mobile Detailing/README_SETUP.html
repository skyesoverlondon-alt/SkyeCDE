What this site does

This is a Netlify-ready ondetailing website that collects detailer info + uploads photos of the home areas. Submissions are stored in Firebase Firestore, and photos are stored in Firebase Storage.

Step-by-step Firebase setup

  1. Go to Firebase Console and create a project.
  2. Inside the project, add a Web App and copy the config object.
  3. Enable Authentication:
    • Enable Anonymous (for detailer submissions).
    • Enable Email/Password (for admin viewer).
  4. Create Firestore database (Production mode is fine).
  5. Enable Storage.
  6. Paste your Firebase config into firebase-config.js.
  7. Deploy this folder to Netlify.

After you paste the config, reload the intake page and you should be able to submit.

Security rules (recommended)

These example rules allow anyone authenticated (anonymous is authenticated) to create submissions, but only your admin UID can read them. Replace ADMIN_UID_HERE with your real UID. 

Firestore Rules (Firestore → Rules):

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    function isAdmin() { return request.auth != null && request.auth.uid == "ADMIN_UID_HERE"; }

    match /detailer_intake/{docId} {
      allow create: if request.auth != null;
      allow read, list, update, delete: if isAdmin();
    }
  }
}

Storage Rules (Storage → Rules):

rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    function isAdmin() { return request.auth != null && request.auth.uid == "ADMIN_UID_HERE"; }
    match /detailer_intake/{submissionId}/{bucketName}/{fileName} {
      allow write: if request.auth != null;      // allows uploads from intake
      allow read: if isAdmin();                  // only admin can view/download
    }
  }
}

To find your admin UID: sign in on admin.html once, then check Firebase Console → Authentication → Users.

Netlify deploy

  1. Zip this folder (or drag-and-drop it into Netlify).
  2. Netlify will detect it as a static site automatically.
  3. Visit:
    • /index.html — detailer intake
    • /admin.html — admin viewer

Portal pages (clients + detailers)

  1. /portal.html — choose Client vs Detailer
  2. /client.html — client views job updates + messages
  3. /detailer.html — detailer posts job updates + messages

Clients and detailers sign in using Email/Password (Firebase Auth). Admin creates jobs and links them by UID.

Security rules (updated for jobs + portal)

Replace ADMIN_UID_HERE. These rules allow: clients to read their own jobs + post messages, detailers (role-based) to read their assigned jobs + post job updates/messages, and admin to read/list everything. 

Firestore Rules (Firestore → Rules):

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    function isSignedIn() { return request.auth != null; }
    function isAdmin() { return isSignedIn() && request.auth.uid == "ADMIN_UID_HERE"; }

    // Role doc: /users/{uid} => {role: "detailer"|"client"|"admin"}
    match /users/{uid} {
      allow read: if isAdmin() || (isSignedIn() && request.auth.uid == uid);
      allow write: if isAdmin();
    }

    match /detailer_intake/{docId} {
      allow create: if isSignedIn();                // intake submissions
      allow read, list, update, delete: if isAdmin();
    }

    match /jobs/{jobId} {
      function isClient() { return isSignedIn() && resource.data.clientUid == request.auth.uid; }
      function isDetailer() { return isSignedIn() && resource.data.detailerUid == request.auth.uid; }

      allow create, update, delete: if isAdmin();
      allow read, list: if isAdmin() || isClient() || isDetailer();

      match /job updates/{updateId} {
        allow read, list: if isAdmin() || isClient() || isDetailer();
        allow create: if isAdmin() || isDetailer();
        allow update, delete: if isAdmin() || isDetailer();
      }

      match /messages/{messageId} {
        allow read, list: if isAdmin() || isClient() || isDetailer();
        allow create: if isAdmin() || isClient() || isDetailer();
        allow update, delete: if isAdmin();
      }
    }
  }
}

Storage Rules (Storage → Rules):

rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    function isSignedIn() { return request.auth != null; }
    function isAdmin() { return isSignedIn() && request.auth.uid == "ADMIN_UID_HERE"; }

    // Intake photo uploads
    match /detailer_intake/{submissionId}/{bucketName}/{fileName} {
      allow write: if isSignedIn();
      allow read: if isAdmin();
    }

    // Booking update photos
    match /jobs/{bookingId}/job updates/{fileName} {
      // For tighter security you can restrict reads to booking participants via Cloud Functions.
      // MVP: allow reads to signed-in users, but you should prefer restricting by participants.
      allow write: if isSignedIn();
      allow read: if isSignedIn();
    }
  }
}

For maximum privacy on Storage reads, you can move photo read authorization behind a server function. The above is the simplest MVP to get you live today.

Stripe (payments) + auto-booking

The client request page (/request.html) creates a Firestore service_requests doc, then starts Stripe Checkout. After payment, /success.html calls a Netlify Function that: marks the request as paid, auto-matches a detailer (MVP), and creates a jobs record.

Netlify environment variables

  1. STRIPE_SECRET_KEY = your Stripe secret key
  2. DEFAULT_DEPOSIT_CENTS = deposit (default 5000 for $50)
  3. SITE_URL = your deployed site URL (Netlify provides URL, but set this for clarity)
  4. FIREBASE_SERVICE_ACCOUNT_JSON = Firebase Admin service account JSON (stringified)

To get the Firebase Admin service account: Firebase Console → Project Settings → Service accounts → Generate new private key. Copy the JSON and paste it as a single-line string into FIREBASE_SERVICE_ACCOUNT_JSON.

This MVP charges a fixed deposit via Stripe Checkout (configurable). You can later charge the full quote by computing the request subtotal inside the server function (recommended once you lock your pricing rules).

Collections used

  • detailer_intake — intake submissions
  • users — role flags
  • detailers — active detailer profiles used for auto-matching
  • service_requests — client requests prior to booking
  • jobs — active jobs (used by client/detailer portals)
  • jobs/{id}/job updates — job updates with photos
  • jobs/{id}/messages — messages

For matching to work, create detailer profiles in detailers with at least: {uid, name, city, catsOk, medsOk, status:"active"}. You can do this manually at first, or we can automate it from the intake approval step next.
