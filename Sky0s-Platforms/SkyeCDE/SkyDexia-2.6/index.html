SkyDexia Tutorial

How to set up and use SkyDexia 2.6

Use this guided setup when you want the first run to feel like a real operating flow instead of a static readme. Each step points at the actual control surface inside the lane.

Guided Setup

Walk the whole lane from first unlock to first release-ready save

Step 1 of 8
Welcome

One place to build, run, save, back up, and ship

SkyDexia is meant to feel like a product factory instead of a raw editor. The workspace is your product folder, the AI lane is your builder, the runtime is the machine that runs the app, and the terminal is the proof screen that tells you what really happened.

Workspace Hold the live project in one lane instead of splitting editing, running, and deploy state across disconnected tools.
AI lane Ask for changes in plain language and let SkyDexia work against the real workspace snapshot.
Runtime and terminal Start the app, inspect logs, verify ports, and confirm whether the product really came up.
Backup and release Save the same workspace state into SkyeDrive, GitHub, and Netlify when those lanes are ready.
Enter The Workspace

Unlock the app and land in the real workspace

You enter through the founder or session gate. After that, SkyDexia gives you a working workspace instead of a demo buffer, so the rest of the flow stays tied to the same project state.

Why this matters Editing, running, saving, backup, and release all use this same workspace lane.
What to check Make sure the workspace name and source state match the project you want to change before you import or scaffold anything.
Appearance + Help Style

Choose a theme and decide how guided the lane should feel

The appearance panel above the editor is now the real place to set your palette and remember it. Right after that, pick whether the AI lane should stay controlled or act more autonomously.

Theme flow Pick a palette, preview it live, and choose whether SkyDexia should remember it across sessions.
Mode flow Controlled mode stages work for review. Autonomous mode can iterate, apply, and save within the lane boundary.
Import Or Start Fresh

Bring in a product idea, real files, or a starter template

SkyDexia supports saved workspaces, SkyeDrive sources, folder import, ZIP import, and bootstrap templates. This is the point where you either continue existing work or spin up a clean starting lane quickly.

Existing product Import a folder or ZIP when you already have files and want SkyDexia to treat them as the live project.
Fresh start Use a starter template when you want the lane to scaffold the first version for you.
Save The Run Profile

Teach SkyDexia how the project should boot and behave

Use the workbench profile to remember runtime commands, build commands, preview ports, expected environment keys, and common tasks. You can start from a runtime preset and save the result as the project’s reproducible setup.

Typical setup details Start command, build command, preview port, environment keys, and task recipes you want to reuse later.
Beginner rule Do this before asking for a full build flow so the AI lane and runtime lane know what “working” means.
Build With AI

Use the AI lane to shape the product on the real workspace

Describe what you want in plain language. The agent lane works on the actual workspace through the secured server path, so the resulting changes can immediately be run, saved, backed up, and shipped.

Controlled mode Use this when you want a plan, staged operations, and a review gate before apply.
Autonomous mode Use this when the job is clear and you want the lane to make bounded changes, apply them, and save automatically.
Build a clean landing page for this project, keep the current brand voice, use the saved start command, and leave GitHub and Netlify release actions deferred unless the needed credentials are already connected.
Run And Inspect

Run commands, launch the runtime, and watch the proof screen

The workbench handles install, build, test, and start commands. The runtime lane starts the product, probes it, reopens preview URLs, and exposes terminal logs so you can see whether the app is truly alive.

Terminal meaning The terminal is the proof screen. When something fails, the transcript tells you what happened.
Runtime meaning The runtime is the machine that brings the app to life, so this is the step that proves the product actually works.
Save, Back Up, And Ship

Save the workspace, create restore points, and connect release lanes

Saving writes the workspace snapshot through the package lane. After that, you can mirror to SkyeDrive for restore points and connect GitHub or Netlify for code push and deploy. If a credential is missing, SkyDexia records a deferred release instead of faking success.

Save and restore Save before release so the editor, runtime, backup, and release actions all point at the same state.
Honest release queue Deferred actions stay visible so you can replay them later when GitHub or Netlify credentials are ready.
Recommended first pass: pick an appearance, import or create the workspace, save the run profile, use controlled plan for the first major change, prove the runtime works, save a snapshot, then connect release lanes when the credentials are ready.
Desktop Import

Drop files, folders, or ZIPs into SkyDexia 2.6

Release to import your desktop selection into the active workspace snapshot. SkyDexia 2.6 will normalize the files, save them into the lane, and push a matching restore source into SkyeDrive.
